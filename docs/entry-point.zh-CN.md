# 为什么先采用 Blackboard

[English](./entry-point.md)

大多数基础设施都要求你先改变工作方式，然后才给你回报。Blackboard 恰恰相反：它是
进入 Octopus 技术栈**成本最低、风险最小的入口**，而且在第二个智能体接触你的代码库
的那一刻就开始产生价值。

## 一分钟理由

- **一条命令。** `npx octopus-blackboard quickstart` 会初始化黑板、检测你的 MCP
  客户端，并给出可直接粘贴的配置。无需账号、无需运行服务、无需设计任何表结构。
- **今天就能用，靠你已有的工具。** 你不需要编排器、调度器或技术栈的其余部分。让
  Claude Code 和 Cursor 指向同一个 `.octoboard/`，它们就已经共享记忆了。这就是
  完整的产品。
- **本地优先、可离线。** 黑板就是 `.octoboard/` 下的单个 SQLite 文件，从当前工作
  目录向上查找（类似 `.git`）。任何数据都不会离开你的机器。读写它无需任何网络请求。
- **纯增量，绝不碍事。** Blackboard **不是**编排器。它不调度智能体、不触发它们、
  也不替它们做任何决定。它只记录与呈现，由智能体决策。你可以随时停用它，删掉一个
  目录即可。

## 第一天你实际得到什么

在同一个代码库上工作的两个智能体通常对彼此一无所知——一个在重构 auth，另一个在
改写同一个文件；上下文散落在各个 CLI 中；没人能回放发生了什么。有了黑板：

- **共享的协调记忆**——谁在做什么、改动了什么、决定了什么、有哪些未解风险、交接给
  了谁。
- **冲突感知**——黑板从不阻塞，而是**呈现**。若两个智能体认领同一个键，或触碰同一
  任务的文件，两次写入都会被记录，并向第二个智能体发出警告。
- **防篡改时间线**——每次写入都会向仅追加日志追加一条哈希链条目。历史可审计、可
  回放，任何事后对早先条目的修改都会破坏校验。

## 时间线是其他工具赖以构建的事实来源

Blackboard 刻意定位为**捕获层**。它自 v0.2.1 起的哲学很简单：*协议传输事实，消费者
派生含义。* 黑板把发生的事情作为朴素、可校验的事实记录在时间线上，而不试图替你解读。

这正是它适合作为**第一个**采用对象的原因。因为时间线是一份中立、防篡改的记录——
而非某种专有工作流——你之后添加的任何东西，都是从一个已然可信的基础上读取。你并
非押注于某个工作流引擎，而只是写下无论今后采用什么都会需要的事实。

## 对路线图保持诚实

明确区分已有功能与规划中的功能：

- **今天已有：** 本地 SQLite 黑板、CLI（`octoboard` / `blackboard`）、任何客户端都
  能连接的 MCP 服务器、哈希链时间线、会话、与 Git 提交挂钩的与厂商无关的 AI/人类
  归因、评审、查询层，以及只读的 `serve` 仪表盘。
- **Beta / 路线图边界：** Postgres sync target 已可用于可携带的归属记录，并有 live
  CI 覆盖；大规模运营与变更订阅仍在加固中。与其他 Octopus 仓库的更深集成——例如与
  **octopus-evidence** 更丰富的桥接，让黑板上捕获的证据流入专门的证据与验证层——是
  **未来的工作，而非当前功能。** 今天 Blackboard 仍然是捕获层：它把证据作为事实记录，
  把含义的派生交给消费者。请不要抱着
  一个尚未构建的证据验证流水线的预期去采用它。

先采用 Blackboard，是因为它**本身**就有用——在你已经运行的工具上，无需安装任何
其他东西。技术栈中的其余一切都是可选的，都是在一个你已经可以信任的时间线之上做
增量。
