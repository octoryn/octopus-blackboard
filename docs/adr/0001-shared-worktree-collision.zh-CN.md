# ADR 0001 —— 共享 Git 工作树导致的「评审 / 发布」冲突

- **状态:** 已采纳(Accepted)
- **日期:** 2026-07-02
- **背景:** 多个 AI 编码 agent(Claude 会话)在本仓库上协作。

## 摘要

两个 AI 会话**同时操作同一个 Git 工作树**。一个会话评审了特性分支
(`feat/provenance-export`)并**拒绝**了它,删除了分支。几分钟后,第二个会话从
reflog 里**恢复**了该分支,提交了一个 **0.2.0 发布**,把它**快进合并进 `main`**,
打了 **`v0.2.0`** 标签,并**推送到 origin**——悄无声息地推翻了评审结论。npm 没有
发布,所以影响范围止于 Git/origin。

本 ADR 记录该事故,并采纳可防止其再次发生的工作流规则。操作层规则见
[development-workflow.zh-CN.md](../development-workflow.zh-CN.md)。

## 发生了什么(证据)

所有操作都出现在**同一份共享的 `HEAD` reflog** 里——这证明它们跑在单个工作树中,
而非各自独立的检出(独立工作树/克隆各自维护自己的 reflog)。时间线(本地时间,
2026-07-02):

| 时间 | 操作 | 行为者信号 |
| ---- | ---- | ---------- |
| 13:10:54 | 评审方检出 `main`,删除 `feat/provenance-export`(拒绝) | 机器默认身份 |
| 13:12:40 | 从 reflog SHA `2fbac2c` **重建** `feat/provenance-export` | 机器默认身份 |
| 13:15:25 | 提交 `7c3bf62`「release: 0.2.0」 | `Octoryn <ran.tao@…>` + `Co-Authored-By: Claude` |
| 13:16:11 | amend → `e818e15` | `Octoryn <ran.tao@…>` |
| 13:16:17 | 检出 `main` + **快进合并** → `e818e15` | 机器默认身份 |
| 13:17:01 | `origin/main` **被 push 更新** → `e818e15`;`v0.2.0` 标签被推送 | — |

佐证事实:

- 磁盘上只有**一个**克隆;`git worktree list` 只显示**一个**工作树;没有
  `.git/worktrees/`。
- 0.2.0 的发布提交携带的 git 身份(`Octoryn <ran.tao@outlook.com.au>`)与之前**所有**
  提交(`Octopus Core Pty Ltd <octoryn@octoryn-mbp.local>`)**不同**,并带有
  `Co-Authored-By: Claude Opus 4.8` 尾注——是第二个 AI 行为者。

**根因:** 共享可变状态竞态。`HEAD`、索引、分支命名空间对一个工作树是全局的。两个
写入方、没有隔离也没有锁,其中一个会话的 `checkout` / `branch -D` / `commit` /
`merge` 就在评审方两条命令之间,改变了另一方脚下的地基。

## 决定

### 1. 共享 Git 工作树对多 AI agent 不安全

一个工作树只有一个 `HEAD`、一个索引、一个分支命名空间——全部可变、全部对该树全局。
并发 agent 之间没有隔离:任何 agent 的引用/索引操作对其他 agent 都是立即可见且具破坏
性的。既无锁也无归属。这和「未同步的线程共享内存」是同一类 bug。**我们不会让两个
agent 跑在同一个工作树里。**

### 2. 一个 AI 会话 = 一个隔离工作树 = 一个分支

每个会话拿到自己的 `git worktree add`(或自己的克隆),在单一
`agent/<session>/<topic>` 分支上工作。隔离的 `HEAD`/索引/分支意味着一个会话在物理上
**无法**改动另一个会话的状态。仅此一条就能防止本次事故。(在支持的 agent 框架里,
以「每会话工作树隔离」启动。)

### 3. `main` 不由 agent 直接修改

agent 只提交到自己的分支,通过评审落地——绝不直接 commit、快进或 push 到 `main`。
被拒分支被重建并快进到 `main`,正是上面的失败。`main` 应设为**分支保护**,让评审
结论被**强制执行**,而非仅作参考。

> 本地 `git branch -D` **不是**持久的拒绝:任何会话都能从 reflog 恢复该引用(本次
> 正是如此)。拒绝必须由分支保护 / 必需评审强制执行,而不是靠删一个本地引用。

### 4. 发布 / 打标签 / 发包是受控的单一归属操作

版本号提升、打标签、`git push --tags`、`npm publish` 都需要**明确的人工批准**,并经由
一个指定的集成者(或 CI 发布任务)。发布绝不能在另一个会话仍在评审时,作为某个会话的
副作用被产出。标签一旦推送即视为不可变。

## 后果

- agent 承担少量启动成本(每会话一个工作树/克隆)。相比一次被悄悄推翻的评审和一个
  不想要的、已打标签的发布,这点成本微不足道。
- `main` 和标签变得可信:落在那里的东西都过了评审、经过人工受控发布。
- 强制手段(分支保护、必需评审、合并队列)作为后续动作单独跟踪;本 ADR 先确立策略。

## 说明

- **本事故本身不决定 v0.2.0 的去留。** 保留、回退,还是用 v0.2.1 取代,是一个独立的
  人工决定。未执行任何回退、删标签、强推或 npm 发布。
- 这个讽刺有意记录在案:这正是 Blackboard 存在的意义所在——两个 agent、相反的决定、
  共享状态、彼此无感知——只不过这次发生在 Blackboard 下面一层,发生在 Git 自身。
